# 01 - 总体架构设计

## 1. 项目概述

AICO-Bot 是一款基于 Electron 的跨平台桌面 AI 助手应用，将 Claude Code SDK 封装为可视化的交互界面，并提供自动化平台（Digital Humans）、远程代理管理、多工作空间等功能。

### 1.1 核心功能

- **AI 对话**: 通过 Claude Agent SDK 提供智能对话能力，支持思考过程展示、工具调用、文件编辑
- **工作空间 (Spaces)**: 多工作目录管理，每个空间独立维护对话历史和配置
- **Digital Humans 自动化平台**: 基于 YAML 规范定义自动化 AI 应用，支持定时触发、文件变更触发、Webhook 触发
- **远程代理 (Remote Agent)**: 通过 WebSocket + SSH 隧道连接远程 Claude 代理
- **文件监控**: 独立 Worker 进程监控文件系统变更，触发自动化流程
- **Artifact 管理**: 可视化查看和编辑 AI 生成的文件
- **OpenAI 兼容路由**: 本地代理提供 OpenAI 格式 API 端点

### 1.2 技术栈

| 层次 | 技术 |
|------|------|
| 框架 | Electron 38.x |
| 构建 | electron-vite + electron-builder |
| 渲染进程 | React 18 + TypeScript + Tailwind CSS + Zustand |
| 主进程 | Node.js + Express + better-sqlite3 |
| AI SDK | @anthropic-ai/claude-agent-sdk |
| 通信 | Electron IPC + WebSocket |
| 测试 | Vitest (单元) + Playwright (E2E) |
| 国际化 | i18next (7 种语言) |

### 1.3 项目版本

- 当前版本: `2.0.2`
- 包名: `aico-bot`
- 代码仓库: `openkursar/hello-aico-bot`

## 2. 多进程架构

### 2.1 进程模型

```
┌─────────────────────────────────────────────────────────────────┐
│                        AICO-Bot Desktop                         │
│                                                                 │
│  ┌─────────────────┐         ┌──────────────────┐              │
│  │   Main Process  │◄──IPC──►│ Renderer Process │              │
│  │   (Node.js)     │         │   (React UI)     │              │
│  │                 │         │                  │              │
│  │  ├─ Express HTTP│         │  ├─ Zustand Store│              │
│  │  ├─ SQLite DB   │         │  ├─ React UI     │              │
│  │  ├─ Claude SDK  │         │  ├─ API Adapter  │              │
│  │  ├─ Platform    │         │  └─ i18n         │              │
│  │  └─ Apps Engine  │        │                  │              │
│  └────────┬────────┘         └──────────────────┘              │
│           │ IPC                                                   │
│           ▼                                                       │
│  ┌─────────────────┐         ┌──────────────────┐              │
│  │   Preload       │         │   Worker Process │              │
│  │   (Bridge)      │         │   (File Watcher) │              │
│  │                 │         │                  │              │
│  │  window.aicoBot │         │  独立子进程        │              │
│  └─────────────────┘         └──────────────────┘              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 进程职责

| 进程 | 职责 | 关键文件 |
|------|------|----------|
| **Main Process** | Electron 主进程，管理窗口、IPC、SDK 调用、数据库、HTTP 服务器、后台任务 | `src/main/index.ts` |
| **Renderer Process** | 用户界面，React 组件、状态管理、API 调用 | `src/renderer/main.tsx` |
| **Preload Script** | 安全桥接，将 IPC 能力暴露给渲染进程 | `src/preload/index.ts` |
| **Worker Process** | 独立子进程，文件系统监控 | `src/worker/file-watcher/index.ts` |

### 2.3 进程间通信

```
Renderer ──invoke──► Main
         ◄──handle──┘

Main ──sendToRenderer──► Renderer
     ◄──ipcRenderer.on──┘

Main ──spawn──► Worker (stdin/stdout 消息传递)
     ◄────────┘
```

## 3. 启动流程

### 3.1 两阶段启动模型

AICO-Bot 采用**两阶段启动模型**，保证快速首屏渲染，同时完成所有后台服务初始化。

```
app.whenReady()
    │
    ├──► Phase 0: 预启动 (异步，有 I/O)
    │     1. electron-log 初始化
    │     2. 全局错误处理器注册
    │     3. GPU 兼容性设置 (仅 Windows)
    │     4. 防指纹识别设置
    │     5. 单实例锁获取
    │     6. 协议注册 (aico-bot-file://)
    │     7. 数据目录创建
    │     8. 实例 ID 初始化
    │     9. 应用菜单创建
    │    10. 主窗口创建 (隐藏状态)
    │
    ├──► Phase 1: 核心服务 (同步，目标 <500ms)
    │     1. 启动状态处理器
    │     2. 配置处理器 (最先注册，其他服务可能依赖)
    │     3. 认证处理器
    │     4. 空间处理器
    │     5. 对话处理器
    │     6. Agent 处理器 (核心功能)
    │     7. Artifact 处理器
    │     8. 系统处理器 (窗口控制)
    │     9. 更新器 + 自动更新检查
    │
    ├──► 窗口显示 (ready-to-show 事件)
    │     mainWindow.show()
    │
    └──► Phase 2: 扩展服务 (异步，不阻塞 UI)
          1. 各扩展 IPC 处理器注册
          2. initBackground() - 后台守护进程
          3. initTerminalGateway() - 终端网关
          4. initPlatformAndApps() - 平台 + 应用引擎
          5. initHealthSystem() - 健康检查
          6. startMcpProxy() - MCP 代理
          7. 发送 bootstrap:extended-ready 到渲染进程
```

### 3.2 阶段依赖保证

- **Phase 1** 完全同步，仅注册 IPC 处理器，无异步 I/O
- **Phase 2** 在 `ready-to-show` + `setImmediate()` 后执行，确保首帧已渲染
- `initBackground()` 必须在 `initPlatformAndApps()` 之前完成（后者依赖前者创建的单例）
- 平台子系统的启动顺序: `initStore()` → `initScheduler/EventBus/Memory` → `scheduler.start()/eventBus.start()`

### 3.3 关闭流程

```
before-quit 事件
    │
    ├── 1. flushAllPendingIndexWrites() - 对话索引持久化
    ├── 2. shutdownHealthSystem() - 标记干净退出
    ├── 3. disableRemoteAccess() - 停止远程访问
    ├── 4. stopOpenAICompatRouter() - 停止 OpenAI 路由
    ├── 5. cleanupExtendedServices() - 反向顺序拆卸
    │     ├── shutdownRegistryService()
    │     ├── stopMcpProxy()
    │     ├── shutdownEventBus()
    │     ├── shutdownScheduler()
    │     ├── shutdownStore()
    │     ├── shutdownBackground()
    │     ├── cleanupAIBrowserHandlers()
    │     ├── cleanupOverlayHandlers()
    │     ├── cleanupSearchHandlers()
    │     ├── shutdownTerminalHistory()
    │     ├── shutdownTerminalGateway()
    │     └── cleanupAllCaches()
    │
    └── 超时保护: 5000ms 强制退出
```

**保活门控**: 如果 `backgroundService.shouldKeepAlive()` 返回 `true`（例如有自动化应用正在运行），`window-all-closed` 事件不会退出进程，而是通过系统托盘保持运行。

## 6. 关键设计原则

### 6.1 模块隔离

- 主进程模块不直接 import 渲染进程模块，通过 IPC 通信
- 共享类型放在 `src/shared/`，确保两端类型一致
- 平台子系统（event-bus, scheduler, memory, store）彼此独立，通过明确的接口交互

### 6.2 延迟加载

- 重模块（如 AI Browser 处理器）在首次调用时才动态 import
- Phase 2 扩展服务不阻塞 UI 渲染
- 数据库连接采用懒打开 + 健康检查复用策略

### 6.3 错误隔离

- 事件总线中每个订阅者的处理函数独立执行，一个抛错不影响其他
- SDK 会话健康监控: 连续失败 5 次后自动重建会话
- 自动化应用连续出错 5 次后自动暂停

### 6.4 安全

- Preload 使用 `contextBridge.exposeInMainWorld` 安全暴露 API
- HTTP 服务器使用 Bearer Token 认证 (默认 6 位 PIN)
- SSH 隧道支持密码和私钥认证
- 文件下载路径遍历保护
