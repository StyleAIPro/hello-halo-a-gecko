# 平台子系统设计

> 源码路径：`src/main/platform/`
> 版本：V1
> 更新时间：2026-04-15

平台子系统为 AICO-Bot 的 Digital Humans 自动化平台提供底层基础设施，包含事件总线（Event Bus）、记忆系统（Memory）、调度器（Scheduler）、持久化存储（Store）和后台服务（Background Service）五大模块。各模块通过明确的接口契约解耦，由 `bootstrap/extended.ts` 统一装配。

---

## 1. 事件总线（Event Bus）

> 源码路径：`src/main/platform/event-bus/`

事件总线是统一的事件路由中枢，连接所有信息源（文件变更、Webhook、定时触发）与自动化 App。它执行过滤和去重，但**不处理事件**，也不感知 AI/LLM。

### 1.1 核心接口

#### 1.1.1 AicoBotEvent

所有事件源产生的标准化事件结构：

```typescript
interface AicoBotEvent {
  id: string              // UUID v4，由总线在 emit 时自动分配
  type: string            // 点分事件类型，如 "file.changed"、"webhook.received"、"schedule.due"
  source: string          // 事件源适配器 ID
  timestamp: number       // Unix 毫秒时间戳，由总线自动分配
  payload: Record<string, unknown>  // 任意载荷数据
  dedupKey?: string       // 可选去重键，同一键在 TTL 窗口内重复则丢弃
}
```

#### 1.1.2 EventFilter

订阅过滤器，所有条件为 AND 逻辑，忽略的条件视为"匹配任意"：

```typescript
interface EventFilter {
  types?: string[]    // 支持 glob 模式："file.*" 匹配所有 file. 前缀类型，"*" 匹配全部
  sources?: string[]  // 精确匹配源适配器 ID
  rules?: FilterRule[]  // 字段级规则匹配
}

interface FilterRule {
  field: string   // 点分字段路径，支持数组索引："payload.items[0].price"
  op: 'eq' | 'neq' | 'contains' | 'matches' | 'gt' | 'lt' | 'in' | 'nin'
  value: unknown  // 比较值
}
```

#### 1.1.3 EventBusService

总线对外公开接口，由 `createEventBus()` 工厂函数创建：

```typescript
interface EventBusService {
  emit(event: Omit<AicoBotEvent, 'id' | 'timestamp'>): void
  on(filter: EventFilter, handler: EventHandler): Unsubscribe
  registerSource(source: EventSourceAdapter): void
  removeSource(sourceId: string): void
  listSources(): EventSourceInfo[]
  start(): void
  stop(): void  // 停止所有源、清空订阅、清空去重缓存
}
```

#### 1.1.4 EventSourceAdapter

统一的信息源适配器接口：

```typescript
interface EventSourceAdapter {
  id: string
  type: EventSourceType  // 'file-watcher' | 'webhook' | 'schedule-bridge' | 'webpage' | 'rss' | 'internal'
  start(emit: EventEmitFn): void   // 开始产出事件，调用 emit 回调推送到总线
  stop(): void                     // 停止产出，清理监听器，必须安全可重入
}
```

### 1.2 过滤引擎

> 源码：`filter.ts`

过滤引擎是"零 LLM 成本"的预过滤层，纯规则匹配，无外部调用。

#### 匹配流程

`matchesFilter(event, filter)` 按顺序检查三个维度（全部 AND 逻辑）：

1. **类型匹配**（数组内 OR）：调用 `matchTypeGlob(event.type, pattern)`
   - `"*"` 匹配所有
   - `"file.*"` 检查 `event.type.startsWith("file.")`
   - `"file.changed"` 精确匹配
2. **源匹配**（数组内 OR）：`filter.sources.includes(event.source)`
3. **规则匹配**（数组内 AND）：逐条执行 `evaluateRule(event, rule)`

#### 字段路径解析

`getByPath(obj, path)` 手写实现的路径解析（无 lodash 依赖）：
- 正则 `/([^.[\]]+)|\[(\d+)\]/g` 解析点号和方括号
- 支持 `"type"`、`"payload.extension"`、`"payload.items[0].name"`
- 遇到 null/undefined 或类型不匹配时返回 `undefined`

#### 操作符语义

| 操作符 | 语义 |
|--------|------|
| `eq` | 严格相等（`===`） |
| `neq` | 严格不等（`!==`） |
| `contains` | 字符串包含子串，或数组包含元素 |
| `matches` | 字符串匹配正则表达式 |
| `gt` | 数值大于 |
| `lt` | 数值小于 |
| `in` | 字段值在目标数组中 |
| `nin` | 字段值不在目标数组中 |

### 1.3 去重缓存

> 源码：`dedup.ts`

内存 TTL 去重缓存，不需要持久化（消费者按幂等设计）。

```typescript
interface DedupCache {
  isDuplicate(key: string | undefined | null, now?: number): boolean
  clear(): void
  size(): number
}
```

**关键参数**：
- `DEFAULT_TTL_MS = 60_000`（60 秒），覆盖 Webhook 重试和文件监听爆发
- `DEFAULT_MAX_SIZE = 10_000`（1 万条目），最坏情况约 1MB 内存
- 使用 ES `Map`（插入顺序保证），`touch(key, now)` 通过 delete + re-set 将条目移至最新位置

**裁剪策略**：
1. 每次 `isDuplicate()` 调用时，先清除超过 TTL 的过期条目（遍历全 Map）
2. 若超过 `maxSize`，从最旧端（Map 迭代器首项）逐条删除

### 1.4 源适配器

#### 1.4.1 FileWatcherSource

> 源码：`sources/file-watcher.source.ts`

桥接 `watcher-host.service.ts` 的 `ProcessedFsEvent` 批处理事件。

**集成方式**：调用 `watcherHost.addFsEventsHandler(callback)` 获取多订阅者支持的回调（避免 handler 覆盖）。

**事件映射**：

| 文件监听 changeType | AicoBotEvent type |
|---------------------|-------------------|
| `add` / `addDir` | `file.created` |
| `change` | `file.changed` |
| `unlink` | `file.deleted` |

**去重键**：`fw:{changeType}:{filePath}`，防止快速重触发。

**Payload 包含**：`spaceId`、`filePath`、`relativePath`、`changeType`、`extension`、`parentDir`、`isDirectory`，以及可选的 `artifact` 和 `treeNode` 数据。

#### 1.4.2 WebhookSource

> 源码：`sources/webhook.source.ts`

在 Express 服务器上挂载 `POST /hooks/:hookPath(*)` 路由。

**安全机制**：
- `/hooks/*` 不在 AICO-Bot 认证中间件之后（外部服务无需 auth token）
- 支持 HMAC-SHA256 签名验证，检查三种标准签名头：
  - `x-hub-signature-256`（GitHub 风格，`sha256=<hex>`）
  - `x-signature-256`（通用）
  - `x-webhook-signature`（备选）
- 使用 `timingSafeEqual` 进行时间安全比较，防止计时攻击
- 请求体限制 256KB（`MAX_BODY_BYTES = 256 * 1024`）
- 签名密钥通过注入的 `WebhookSecretResolver` 回调动态解析

**事件结构**：
- type: `"webhook.received"`
- source: `"webhook"`
- payload: `{ path, body, headers, query, method, ip }`
- dedupKey: 取自 body 的 `dedupKey` 字段，否则生成 `wh:{path}:{body-hash 前16位}`

#### 1.4.3 ScheduleBridgeSource

> 源码：`sources/schedule-bridge.source.ts`

桥接调度器的 `jobDue` 事件到统一事件总线。

**最小接口依赖**：

```typescript
interface SchedulerLike {
  onJobDue(handler: (job: ScheduledJobInfo) => void): void
}
```

支持延迟装配（`setScheduler()`），允许事件总线在调度器就绪前启动。

**事件结构**：
- type: `"schedule.due"`
- source: `"schedule-bridge"`
- payload: `{ jobId, jobName, metadata, scheduledAt }`
- 无 dedupKey（调度器已保证单次触发语义）

## 2. 记忆系统（Memory）

> 源码路径：`src/main/platform/memory/`

记忆系统使 AI 智能体能够在会话之间保持持久知识，采用基于 Markdown 文件的存储方式，配合作用域隔离和权限矩阵。

### 2.1 双层结构

记忆文件 `memory.md` 包含两个顶层部分：

```
# now                          ← 工作记忆（每次运行自动预加载）
## State | 一行摘要            ← 始终在第一，每次运行更新
## [实体名称]                   ← 按实体跟踪（可选）
## Patterns                    ← 学到的规则（累积）
## Errors                      ← 故障记录（压缩）

# History                      ← 时间线（最新在前）
## YYYY-MM-DD-HHmm | 摘要      ← 每次运行一条
### details heading            ← 可选，用于重要事件
```

`# now` 是工作记忆，在每次触发时通过快照构建器自动注入初始用户消息。`# History` 是运行时间线，系统在每次运行前预插入时间戳标题。

### 2.2 作用域类型

```typescript
type MemoryScopeType = 'user' | 'space' | 'app'
```

| 作用域 | 文件路径 | 说明 |
|--------|----------|------|
| `user` | `{spaceDataDir}/user-memory.md` | 全局用户偏好，跨空间 |
| `space` | `{spacePath}/.aico-bot/memory.md` | 每空间知识 |
| `app` | `{spacePath}/.aico-bot/apps/{appId}/memory.md` | 每 App 私有记忆 |

**归档目录**：每个作用域还有 `memory/` 子目录用于存储压缩归档和会话摘要：
- user 作用域：`{spaceDataDir}/user-memory/`
- space/app 作用域：`{basePath}/memory/`
- 运行记录存储在 `memory/run/` 子目录中

### 2.3 权限矩阵

> 源码：`permissions.ts`

```
调用者     作用域     读取    写入(replace)   写入(append)
------     -----     ----    --------------   -------------
user       user      YES     YES              YES
user       space     YES     YES              YES
user       app       NO      NO               NO
app(A)     user      YES     NO               NO
app(A)     space     YES     NO               YES
app(A)     app(A)    YES     YES              YES
app(A)     app(B)    NO      NO               NO
```

权限在两个层面强制执行：
1. **MCP 工具 Schema 层**：通过 `getReadableScopes()` 和 `getWritableScopes()` 动态生成 Zod 枚举，在工具参数层面限制可见选项
2. **运行时层**：`assertReadPermission()`、`assertWritePermission()`、`assertListPermission()` 在实际 I/O 前检查，失败时抛出 `MemoryPermissionError`

### 2.4 文件操作

> 源码：`file-ops.ts`

#### 读取模式

```typescript
type MemoryReadMode = 'full' | 'headers' | 'section' | 'tail'
```

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `full` | 返回整个文件 | 小文件或需要全部内容时 |
| `headers` | 仅返回 Markdown 标题行（带行号） | 低 token 成本了解文件结构 |
| `section` | 返回标题文本匹配的部分内容 | 按需加载特定段落 |
| `tail` | 返回最后 N 行（默认 50） | 查看最近的追加内容 |

**节提取**（`readMemorySection`）：匹配标题后，提取从该标题到下一个同级或更高级别标题之间的内容。匹配方式为不区分大小写的子串匹配。

#### 写入模式

- **`append`**：追加到文件末尾，自动添加审计元数据 `<!-- {ISO时间} by {source} -->`
- **`replace`**：原子替换整个文件（写入临时文件后 `rename`），避免写入中途崩溃导致文件损坏

#### 归档操作

`archiveMemoryFile(filePath, archiveDir)`：
- 将当前 `memory.md` 移动到归档目录
- 归档文件名格式：`YYYY-MM-DD-HHmm.md`
- 同名冲突处理：追加秒数后缀
- 自动确保归档目录存在

### 2.5 快照构建器

> 源码：`snapshot.ts`

`buildMemorySnapshot(caller)` 构建记忆文件的结构化快照，用于两个场景：

1. **触发时注入**：自动化运行前将快照注入初始用户消息，使 AI 即时获得记忆上下文
2. **运行时 `memory_status` 工具**：返回相同结构元数据（不含内容），供 AI 在运行中重新检查记忆映射

**MemorySnapshot 结构**：

```typescript
interface MemorySnapshot {
  exists: boolean              // memory.md 是否存在
  totalLines: number           // 总行数
  sizeBytes: number            // 文件大小（字节）
  firstSection: string | null  // 第一个顶级节的全文（含所有子标题）
  headers: HeadingEntry[]      // 所有标题（行号、层级、节行数）
  fullContent: string | null   // 小文件（<=30行）的完整内容
  archiveFiles: string[]       // 最近 5 个运行记录文件名
  archiveTotalCount: number    // 运行记录文件总数
  compactionArchiveCount: number  // 压缩归档文件数
  memoryFilePath: string       // memory.md 绝对路径
  memoryArchiveDir: string     // memory/run/ 目录路径
  lastModified: string | null  // 最后修改时间（ISO）
  rawContent: string | null    // 原始文件内容（内部用）
}
```

**小文件阈值**：`SMALL_MEMORY_LINE_THRESHOLD = 30` 行，低于此值的文件自动注入全文。

**标题解析**：`parseHeadings(lines)` 计算每个标题对应的节行数，从该标题行到下一个同级或更高级别标题之间。

### 2.6 MCP 工具

> 源码：`tools.ts`

通过 `createMemoryMcpServer(caller, service)` 创建三个 MCP 工具：

| 工具 | 说明 |
|------|------|
| `memory_read` | 读取记忆，支持四种读取模式和指定归档文件 |
| `memory_write` | 写入记忆，支持 append/replace 模式 |
| `memory_list` | 列出归档目录中的文件 |

工具可用的 scope 枚举和 mode 枚举根据调用者身份动态生成（Zod schema 级约束）。

此外，`createMemoryStatusMcpServer(caller)` 提供 `memory_status` 工具，仅返回结构元数据（不含内容），引导 AI 使用原生 Read/Edit/Write 工具操作内容。

### 2.7 压缩机制

压缩阈值：`COMPACTION_THRESHOLD_BYTES = 100 * 1024`（100KB），基于文件大小。

**生命周期**：
1. `needsCompaction(caller, scope)` — 检查文件大小是否超过阈值
2. `compact(caller, scope)` — 将当前 `memory.md` 归档到 `memory/YYYY-MM-DD-HHmm.md`，返回 `{ archived, needsSummary: true }`
3. 调用者（apps/runtime）负责通过 LLM 生成压缩摘要并写入新的 `memory.md`
4. `flushBeforeCompaction(caller)` — V1 中为占位符（无写后缓冲），V2 将用于刷新待写缓存
5. `saveSessionSummary(caller, scope, params)` — 会话结束时将摘要保存到 `memory/run/` 目录

**安全路径解析**：`resolveArchivePath()` 在解析归档文件路径后检查 `resolved.startsWith(archiveDir + sep)`，防止目录遍历攻击。

## 3. 调度器（Scheduler）

> 源码路径：`src/main/platform/scheduler/`

调度器是通用的作业调度引擎，不感知 AI/LLM 或 App。通过 `metadata` 字段透传消费者专属数据。

### 3.1 调度类型

```typescript
type ScheduleKind = 'every' | 'cron' | 'once'
```

| 类型 | 结构 | 说明 |
|------|------|------|
| `every` | `{ kind: 'every', every: "30m" }` | 固定间隔，锚点网格对齐，避免离线后追赶风暴 |
| `cron` | `{ kind: 'cron', cron: "0 9 * * *", timezone?: string }` | 标准 cron 表达式，支持 IANA 时区 |
| `once` | `{ kind: 'once', once: number }` | 一次性执行，执行后自动禁用 |

**间隔解析**（`parseEveryString`）：
- 支持格式：`"30s"`、`"5m"`、`"2h"`、`"1d"`、`"1.5h"`
- 最小间隔：`MIN_INTERVAL_MS = 10_000`（10 秒），防止 CPU 过载

**锚点网格对齐**（`computeNextRunEvery`）：
- 运行网格：`anchor, anchor + every, anchor + 2*every, ...`
- `computeNextRun()` 始终返回严格在 `now` 之后的第一个网格点
- 进程离线多个周期后，仅返回下一个未来网格点，不会补执行

**Cron 计算**：使用 `croner` 库（零依赖 cron 解析器），支持 5 段和 6 段 cron 表达式。

### 3.2 作业生命周期与状态

```typescript
type JobStatus = 'idle' | 'running' | 'paused' | 'disabled'
```

| 状态 | 说明 |
|------|------|
| `idle` | 等待下次运行时间 |
| `running` | 正在执行（handler 已调用，等待结果） |
| `paused` | 用户显式暂停，恢复前不会触发 |
| `disabled` | 因连续错误自动禁用，需显式恢复 |

**运行结果**：
```typescript
type RunOutcome = 'useful' | 'noop' | 'error' | 'skipped'
```

**作业定义**（`SchedulerJob`）关键运行状态字段：
- `anchorMs` — `every` 调度的网格原点时间戳
- `nextRunAtMs` — 计算的下次运行时间（epoch ms）
- `lastRunAtMs` — 上次完成运行的时间戳
- `runningAtMs` — 当前执行开始时间（null 表示未运行）
- `consecutiveErrors` — 连续执行错误计数（成功时重置）

### 3.3 SQLite 持久化

> 源码：`store.ts`

**命名空间**：`scheduler`

#### 表结构

**scheduler_jobs**：

| 列名 | 类型 | 说明 |

| `anchor_ms` | INTEGER | 锚点时间戳 |
| `next_run_at_ms` | INTEGER | 下次运行时间 |
| `last_run_at_ms` | INTEGER | 上次运行时间（可空） |
| `running_at_ms` | INTEGER | 当前运行开始时间（可空） |
| `consecutive_errors` | INTEGER | 连续错误计数 |
| `status` | TEXT | 作业状态 |
| `metadata_json` | TEXT | 元数据 JSON（可空） |
| `created_at` | INTEGER | 创建时间 |
| `updated_at` | INTEGER | 最后修改时间 |

**scheduler_run_log**：

| 列名 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK AUTOINCREMENT | 日志条目自增 ID |

### 3.4 定时器循环

> 源码：`timer.ts`

使用递归 `setTimeout`（非 `setInterval`）实现精确调度控制。

**关键常量**：
```typescript
MAX_TIMER_DELAY_MS = 60_000          // 最大定时器延迟 60 秒
STUCK_JOB_THRESHOLD_MS = 7200000     // 卡住作业阈值 2 小时
MAX_CONSECUTIVE_ERRORS = 5           // 连续错误自动禁用阈值
```

**定时器臂设置**（`armTimer()`）：
1. 清除当前定时器
2. 调用 `findNextWakeTime()` 查找所有启用且非运行中作业的最早 `nextRunAtMs`
3. 计算延迟 `delay = nextWakeMs - now`
4. 钳位到 `MAX_TIMER_DELAY_MS`（确保时钟跳变恢复，并避免 Node.js `setTimeout` 32 位溢出——最大约 24.8 天）

**Tick 执行流程**（`onTick()`）：
1. 并发保护：若上一个 tick 仍在执行，直接重新臂定定时器并返回
2. `cleanupStuckJobs(now)` — 清理卡住的作业
3. `findDueJobs(now)` — 查找已到期的作业（enabled && status==='idle' && !runningAtMs && nextRunAtMs <= now）
4. 顺序执行到期作业
5. `finally` 中设置 `running = false` 并重新臂定定时器

**并发保证**：调度器保证同一作业的 handler 不会被并发调用。

### 3.5 启动恢复

启动时（`start()`）按顺序执行：

1. **清除卡住标记**：`clearStaleRunningMarkers()` 将 `running_at_ms IS NOT NULL AND status = 'running'` 的作业重置为 `running_at_ms = NULL, status = 'idle'`
2. **重算下次运行时间**：`recomputeAllNextRuns(now)` 对所有启用作业重算 `nextRunAtMs`（仅当缺失或为 0 时）
3. **补执行遗漏作业**：`runMissedJobs(now)` 查找已到期但未执行的作业（最多每个作业一次）
4. **臂定定时器**

### 3.6 错误退避

```typescript
ERROR_BACKOFF_MS = [
  30_000,       // 第 1 次错误 → 30 秒
  60_000,       // 第 2 次错误 → 1 分钟
  300_000,      // 第 3 次错误 → 5 分钟
  900_000,      // 第 4 次错误 → 15 分钟
  3600000,      // 第 5 次及以后 → 60 分钟
]
```

**退避应用**（`applyJobResult`）：
- 错误时：`nextRunAtMs = max(normalNext, finishedAt + backoffDelay)`
- 正常时：`nextRunAtMs = normalNext`
- 连续错误达到 `MAX_CONSECUTIVE_ERRORS`（5）时，作业自动禁用（`enabled = false, status = 'disabled'`）
- `once` 类型作业在任何执行后自动禁用

### 3.7 卡住作业检测

`cleanupStuckJobs(now)` 在每次 tick 中执行：
- 检查 `runningAtMs` 非空的作业
- 若 `now - runningAtMs > STUCK_JOB_THRESHOLD_MS`（2 小时），视为卡住（进程崩溃）
- 清除 `runningAtMs` 标记，重置状态为 `idle`
- 重新计算 `nextRunAtMs`

## 4. 存储/数据库（Store）

> 源码路径：`src/main/platform/store/`

集中管理 SQLite 数据库连接、PRAGMA 调优、迁移执行和损坏恢复。

### 4.1 DatabaseManager 单例

> 源码：`database-manager.ts`

通过 `createDatabaseManager(appDbPath)` 创建，内部维护 `Map<string, Database>` 连接缓存。

```typescript
interface DatabaseManager {
  getAppDatabase(): Database.Database       // 获取应用级数据库
  getSpaceDatabase(spacePath: string): Database.Database  // V1 抛出"未实现"
  runMigrations(db, namespace, migrations): void           // 执行迁移
  transaction<T>(db, fn): T                 // 事务执行
  closeAll(): void                          // 关闭所有连接
}
```

**连接健康检查**：每次 `getOrOpen(dbPath)` 时执行 `SELECT 1` 验证连接可用性。若连接已损坏，则移除缓存、关闭旧连接并重新打开。

### 4.2 PRAGMA 设置

新打开数据库时自动应用：

```sql
PRAGMA journal_mode = WAL      -- 写前日志模式：读写不阻塞
PRAGMA synchronous = NORMAL    -- WAL 模式下的最佳性能/持久性平衡
PRAGMA foreign_keys = ON       -- 启用外键约束
PRAGMA busy_timeout = 5000     -- 锁等待超时 5 秒（而非立即失败）
```

### 4.3 损坏恢复

> `openDatabase(dbPath)` 函数

1. 尝试打开数据库，执行 `SELECT count(*) FROM sqlite_master` 完整性检查
2. 若失败且匹配损坏模式，执行恢复：
   - 将损坏文件重命名为 `{name}.corrupt.{timestamp}`
   - 同时重命名 `-wal` 和 `-shm` 辅助文件
   - 创建全新数据库
3. 非损坏错误（EACCES、ENOSPC 等）直接抛出

**损坏检测模式**：
- `SQLITE_CORRUPT`
- `SQLITE_NOTADB`
- `database disk image is malformed`
- `file is not a database`
- `file is encrypted or is not a database`

### 4.4 迁移系统

> `runMigrations(db, namespace, migrations)`

**命名空间隔离**：每个模块（scheduler、app_manager 等）使用独立的命名空间，各自维护版本号序列（从 1 开始）。

**迁移表结构**（`_migrations`）：

| 列名 | 类型 | 说明 |
|------|------|------|
| `namespace` | TEXT PK | 模块命名空间 |
| `version` | INTEGER | 当前版本 |
| `applied_at` | INTEGER | 应用时间戳 |

**执行流程**：
1. 创建 `_migrations` 表（`CREATE TABLE IF NOT EXISTS`，幂等）
2. 读取当前版本（无名空间时为 0）
3. 校验：版本必须为正整数，不可重复
4. 按版本号升序排序
5. 过滤出 `version > currentVersion` 的待执行迁移
6. **单次事务**执行所有待执行迁移
7. 更新 `_migrations` 表中的版本号为最终版本
8. 若任何迁移失败，整个批次回滚

**版本间隙处理**：若存在间隙（如当前 v2，下一个可用 v4），仅输出 WARN 日志，不报错——这在清理旧迁移时是常见模式。

## 5. 后台服务（Background Service）

> 源码路径：`src/main/platform/background/`

管理进程保活、系统托盘和守护浏览器窗口，为自动化任务提供后台运行能力。

### 5.1 保活管理器（Keep-Alive）

> 源码：`keep-alive.ts`

跟踪阻止 Electron 进程在所有窗口关闭时退出的原因。

**关键参数**：
```typescript
MAX_KEEP_ALIVE_TTL_MS = 86400000  // 24 小时
```

**核心类**：`KeepAliveManager`

```typescript
class KeepAliveManager {
  register(reason: string): Unsubscribe    // 注册保活原因，返回注销函数
  shouldKeepAlive(): boolean               // 检查是否有未过期的原因
  getActiveCount(): number                 // 活跃原因数
  getActiveReasons(): string[]             // 活跃原因列表
  clearAll(): void                         // 清空所有原因
}
```

**惰性裁剪**：`shouldKeepAlive()` 调用时执行 `pruneExpired()`，遍历删除超过 TTL 的条目。不依赖后台定时器。

**防重复注册**：同一原因再次注册时刷新时间戳，允许调用者"续期"。

**注销安全**：返回的 `Unsubscribe` 函数带有 `disposed` 标记，防止重复注销。

### 5.2 托盘管理器（Tray）

> 源码：`tray.ts`

管理系统托盘图标和上下文菜单，提供视觉状态反馈和快捷操作。

**平台差异**：

| 平台 | 图标 | 行为 |
|------|------|------|
| macOS | `trayTemplate.png`（模板图片，自适应明暗栏） | 点击显示菜单 |
| Windows/Linux | `tray-16.png`（16x16 PNG） | 点击显示主窗口 |

**图标路径解析**：
- 生产环境：`{process.resourcesPath}/tray/`
- 开发环境：`{app.getAppPath()}/resources/tray/`

**菜单项**：
- `Show AICO-Bot` — 显示主窗口
- `Go Online` / `Go Offline` — 切换在线/离线状态
- `Status: Online/Offline` — 只读状态显示
- `Active Tasks (N)` — 显示最多 5 个活跃保活原因（超过时显示 "... and N more"）
- `Quit AICO-Bot` — 退出应用

**回调委托**：`TrayCallbacks` 接口将业务逻辑委托给 `BackgroundService`，托盘本身不实现业务逻辑。

### 5.3 守护浏览器窗口

> 源码：`daemon-browser.ts`

提供单个共享的隐藏 BrowserWindow 供自动化任务使用。

**窗口配置**：
```typescript
{
  show: false,
  width: 1920, height: 1080,
  webPreferences: {
    partition,           // persist:automation-{domain}
    sandbox: true,
    contextIsolation: true,
    nodeIntegration: false,
    webSecurity: true,
    allowRunningInsecureContent: false
  }
}
```

**会话隔离**：使用 `persist:automation-{domain}` 分区实现域名级 Cookie/Session 隔离。

**FIFO 队列**：
- V1 设计：单窗口，顺序访问
- 窗口正在使用时，新请求进入 `QueueEntry[]` 队列
- 释放时依次处理队列中的下一个请求（`drainQueue()` 递归处理连续失败场景）

**安全定时器**：`AUTO_RELEASE_TIMEOUT_MS = 5 * 60 * 1000`（5 分钟）。若调用者超时未释放，自动释放窗口防止永久阻塞队列。

**隐身脚本注入**：通过 `injectStealthScripts(webContents)` 使用 CDP 注入反检测脚本。注入失败不影响窗口正常使用（best-effort）。

**窗口销毁安全**：
- `destroy()` 设置 `isShuttingDown` 标志
- 拒绝所有队列中的请求
- 销毁窗口并清空所有状态

### 5.4 分区提取（Partition Extraction）

> 源码：`partition.ts`

从 URL 中提取主域名并生成 Electron 会话分区字符串 `persist:automation-{domain}`。

**主域名提取**（`extractMainDomain`）：
- 去除前导 `www.`
- IPv4/IPv6 地址直接使用
- 单标签（如 `localhost`）或双标签（如 `jd.com`）直接使用
- 双部分 TLD 处理：检查最后两部分是否在 `TWO_PART_TLDS` 集合中，若是则取最后三部分

**双部分 TLD 集合**（覆盖常见自动化目标）：
```
co.uk, co.jp, co.kr, co.in, co.nz, co.za, co.id, co.th
com.au, com.br, com.cn, com.hk, com.mx, com.my, com.sg
com.tw, com.ar, com.tr, com.co, com.ph, com.pk, com.pe
com.vn, com.ua, com.ng, com.eg, com.sa, com.bd
net.au, net.br, net.cn, net.nz
org.au, org.uk, org.cn, org.nz, org.hk, org.tw
ac.uk, ac.jp, ac.kr, ac.in
gov.uk, gov.au, gov.cn, gov.in
edu.au, edu.cn, edu.hk
ne.jp, or.jp, go.jp
```

**分区提取示例**：

| URL | 分区 |
|-----|------|
| `https://item.jd.com/12345` | `persist:automation-jd.com` |
| `https://www.taobao.com/search` | `persist:automation-taobao.com` |
| `https://shop.example.co.uk/foo` | `persist:automation-example.co.uk` |
| `http://192.168.1.1:8080/api` | `persist:automation-192.168.1.1` |
| 无效 URL | `persist:automation-unknown`（fallback） |

### 5.5 在线/离线状态管理

```typescript
type BackgroundStatus = 'online' | 'offline'
```

- `goOnline()` / `goOffline()` — 切换状态，通知所有订阅者，更新托盘菜单
- `onStatusChange(handler)` — 订阅状态变化事件
- 状态变更通过 `notifyStatusChange()` 遍历通知所有 handler（带 try/catch 错误隔离）

### 5.6 统一服务接口

`initBackground()` 创建 `BackgroundService` 单例，装配所有子系统：

```typescript
interface BackgroundService {
  // 托盘
  initTray(): void

  // 保活
  shouldKeepAlive(): boolean
  registerKeepAliveReason(reason: string): Unsubscribe

  // 守护浏览器
  getDaemonBrowserWindow(url: string): Promise<BrowserWindow>
  releaseDaemonBrowserWindow(): void

  // 在线/离线
  getStatus(): BackgroundStatus
  goOnline(): void
  goOffline(): void
  onStatusChange(handler: StatusChangeHandler): Unsubscribe
}
```

**关闭流程**（`shutdownBackground()`）：
1. 销毁守护浏览器（拒绝队列请求）
2. 清空所有保活原因
3. 销毁托盘
4. 清空状态变更处理器

## 6. 模块间装配关系

```
bootstrap/extended.ts
    │
    ├── initStore(appDbPath)          → DatabaseManager
    │       └── runMigrations("scheduler", ...)
    │
    ├── initMemory()                  → MemoryService
    │
    ├── initBackground()              → BackgroundService
    │       ├── KeepAliveManager
    │       ├── TrayManager
    │       └── DaemonBrowserManager
    │
    ├── createEventBus()              → EventBusService
    │       ├── registerSource(FileWatcherSource)
    │       ├── registerSource(WebhookSource)
    │       └── registerSource(ScheduleBridgeSource)
    │
    ├── initScheduler()               → SchedulerService
    │       └── SchedulerStore (使用 DatabaseManager)
    │
    └── eventBus.start()              → 启动所有源适配器
```

所有平台模块通过明确的接口契约解耦。消费者只导入接口类型（如 `BackgroundService`、`MemoryService`），不直接访问内部模块。
